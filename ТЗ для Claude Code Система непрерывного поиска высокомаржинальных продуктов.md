## 0. Преамбула для агента

Ты — старший fullstack-разработчик и AI-инженер. Тебе поручено спроектировать и реализовать **внутреннюю агентную систему непрерывного поиска и оценки гипотез по новым высокомаржинальным продуктам** для B2B-химической компании. Стартовый домен — **ЛКМ (лакокрасочные материалы)**, архитектурно поддерживаются ещё **9 направлений**: СОЖ, смазочные материалы, антикоррозийные покрытия, герметики, адгезивы, специальная химия, реагенты, добавки и присадки, поверхностно-активные вещества.

Система **не заменяет** работу R&D-отдела и комитета — она работает как «исследовательский конвейер»: непрерывно сканирует мировые источники (патенты, научные статьи, новости отрасли, рынки сырья), формирует гипотезы по новым продуктам, прогоняет их через пайплайн из специализированных AI-агентов (исследователь → инженер-технолог → маркетолог → финансист → комплаенс → методист → адвокат дьявола) и выводит на комитет уже проработанные предложения с обоснованиями.

**Жёсткие правила:**

1. **Никакой геймификации**: бейджи, очки, уровни, конфетти, эмодзи в продуктовой части — запрещены.
2. **Каждая фича решает реальную задачу пользователя.** Если ты сомневаешься, нужна ли фича — выкидывай.
3. **AI как ядро, а не как маркетинг.** Если задачу можно решить простой функцией, не зови LLM ради звания «AI-driven».
4. **Прозрачность каждого решения.** Любой вывод агента сопровождается источниками и цепочкой рассуждений. Никаких «чёрных ящиков».
5. **Human-in-the-loop переключаемый.** Должен быть тумблер на уровне всей системы и на уровне каждого этапа: «требовать подтверждение человека» / «работать автономно».
6. **Минимум 2–3 клика до любой важной информации.**

Несмотря на правило 1, **визуальная подача данных может и должна быть выразительной**, потому что это инструмент исследования, а не CRUD-форма. Граф знаний, карта процесса, mission control — это функциональные визуализации, а не декорации.

Если тебе кажется, что какое-то решение «ради вау-эффекта», — выкидывай. Если оно реально помогает понять состояние системы быстрее — оставляй.

## 1. Контекст и пользователи

### 1.1 Бизнес-контекст

- B2B-химическая компания, производит ЛКМ (стартовый домен) и смежные продукты.
- Цель системы: непрерывно искать высокомаржинальные ниши и формировать обоснованные гипотезы новых продуктов.
- Источник стратегии: документ «Проект разработки системы для непрерывного поиска и внедрения высоко маржинальных/голубых продуктов», 27.08.2025, Патрушев Е.А.
- 8 этапов воронки: мониторинг трендов → анализ применимости на оборудовании → маркетинговое исследование → расчёт маржи и себестоимости → проверка применимости и стандартов → формирование гипотезы → подготовка к комитету → решение комитета.
- KPI: сокращение времени до новой гипотезы, рост числа гипотез в работе, рост числа гипотез, доведённых до пилота, рост маржи по новым продуктам.

### 1.2 Пользователи (≤ 5 человек)

Маленькая команда, поэтому ролевая модель упрощённая, но **расширяемая**.

1. **Исследователь / Руководитель проекта** — основной пользователь. Запускает поиск, оценивает гипотезы, общается с агентами, принимает решения о продвижении гипотез по этапам.
2. **Инженер-технолог** — даёт обратную связь по применимости на текущем оборудовании, корректирует выводы агентов о технической реализуемости.
3. **Маркетолог / Коммерческий директор** — оценивает рыночную привлекательность, корректирует прогнозы спроса.
4. **Финансист / Экономист** — проверяет расчёты себестоимости и маржинальности.
5. **Член комитета / ЛПР** — read-only + право голоса по гипотезам, представленным на комитет.

Дополнительно: **Администратор** (может совпадать с Руководителем проекта) — управляет агентами, настройками, источниками, интеграциями.

Каждая роль видит свой стартовый дашборд. Любой пользователь может быть назначен «куратором» конкретной гипотезы.

### 1.3 Сценарии использования

- **Утренний обзор:** Исследователь заходит → видит, что за ночь система проработала 47 источников, выдвинула 3 новые гипотезы и обновила оценки 12 существующих → за 15 минут просматривает что важно.
- **Глубокое погружение:** выбирает гипотезу «биоразлагаемые промышленные краски на базе соевого масла» → видит её Hypothesis DNA, цепочку рассуждений каждого агента, источники, исторические оценки → корректирует параметры и переоценивает.
- **Срочный запрос:** руководство просит «ищите альтернативу импортному пигменту X» → активирует **War Room Mode** на эту гипотезу, все агенты приоритизируют её.
- **Подготовка комитета:** выбирает 5 готовых гипотез → в один клик генерирует подборку отчётов и презентацию → комитет голосует прямо в системе.

## 2. Технологический стек

### 2.1 Фронтенд
- **Next.js 15** (App Router) + **TypeScript strict**.
- **TailwindCSS** + **shadcn/ui** (Radix-based) — базовая дизайн-система.
- **TanStack Query** (data fetching) + **TanStack Table** (таблицы с виртуализацией).
- **Zustand** для лёгкого UI-стейта.
- **react-hook-form + zod** для форм.
- **React Flow** (xyflow) — для интерактивной карты процесса и графа знаний.
- **Recharts** + **Visx** — для графиков. Для сложных диаграмм (sankey, force-directed) — **D3** напрямую.
- **Framer Motion** — только для функциональных переходов (раскрытие панелей, transition между состояниями), не для декора.
- **Cmdk** — для command palette.

### 2.2 Бекенд
- **Python 3.12 + FastAPI** — основной API. Python выбран потому, что AI-стек (LangGraph, LlamaIndex, OCR, NLP) на Python качественнее.
- **SQLAlchemy 2** + **Alembic** для миграций.
- **Pydantic v2** для схем.
- **Celery** или **arq** для фоновых задач. Я склоняюсь к **arq** — легче, async-native, хватит для нашей нагрузки.
- **Redis** — кэш, очереди, состояние агентов.

### 2.3 Хранилище данных
- **PostgreSQL 16** с расширением **pgvector** — основная БД + векторный индекс эмбеддингов в одной системе, без отдельной vector DB.
- **Supabase** для MVP (managed Postgres + Auth + Realtime + Storage + Edge Functions). Авторизация через Supabase Auth.
- **S3-совместимое хранилище** для документов (Supabase Storage на старте, Yandex Object Storage при миграции).
- **Schema-driven design**: каждая сущность описана в Pydantic + SQLAlchemy + zod (TypeScript) из единого источника. Используем `datamodel-codegen` или OpenAPI-генерацию для синхронизации.

### 2.4 AI-слой
- **Anthropic Claude** как основная LLM. Через `anthropic` SDK для Python.
- Модели: **Claude Sonnet** для большинства задач, **Claude Opus** для сложного анализа гипотез и финальных заключений (через переключатель на уровне агента).
- **LangGraph** — оркестратор агентов (state machine на python с поддержкой Anthropic).
- **Embeddings:** локальные `bge-m3` или `intfloat/multilingual-e5-large` через `sentence-transformers`. На старте — Supabase pgvector с этими эмбеддингами. Никаких OpenAI Embeddings (нам нужна приватность и поддержка русского).
- **Web research:** для AI-парсинга источников без официальных API используем **Anthropic Claude с web_search и web_fetch tools** (если работаем через API с web tools), либо **Firecrawl** / **ScrapingBee** / **Browserless** как fallback.
- Все вызовы LLM проходят через **AI-Gateway** (собственный модуль), который:
  - Логирует каждый вызов (prompt, response, tokens, cost, latency, agent_id, hypothesis_id).
  - Применяет температуру и параметры из настроек агента.
  - Имеет circuit breaker и retry с exponential backoff.
  - Поддерживает streaming ответов.
  - Считает стоимость в реальном времени (для бюджетного контроля).

### 2.5 Интеграции
- **Битрикс24** — через REST API (адаптер). Двусторонняя синхронизация: гипотезы можно превращать в задачи/сделки в Битрикс24, обратные связи возвращаются.

### 2.6 Инфраструктура

## 3. Архитектура агентной системы (ядро продукта)

Это **самая важная часть документа**. Реализуй её внимательно.

### 3.1 Концепция

Не «один большой LLM-чат». Не «жёсткий ETL-пайплайн». Это **многоагентная система** на LangGraph, где:

- Каждый агент — это **специализированная роль** с своим системным промптом, набором инструментов (tools), температурой, моделью и зоной ответственности.
- Агенты **общаются между собой** через структурированные сообщения (Pydantic-схемы), а не через свободный текст.
- **State** всей системы хранится в БД, не в памяти. Любой workflow можно прервать и продолжить.
- Каждое решение агента — **трассируемое**: видны входные данные, инструкция, цепочка рассуждений, использованные инструменты, источники, выходные данные, токены, стоимость, время.

### 3.2 Состав агентов

Реализуй следующих 9 агентов. Каждый — отдельный класс в `packages/agents/`, наследник `BaseAgent`. Каждый имеет описание роли, allowed_tools, default_temperature, default_model, и system_prompt в `packages/prompts/`.

| # | Агент | Роль и задача | Default temp | Default model |
|---|-------|---------------|--------------|---------------|
| 1 | **Scout** (Разведчик) | Непрерывный мониторинг источников: патенты, научные статьи, новости отраслевые, рынки сырья. Формирует **сырые сигналы** (raw signals) с метаданными источника. | 0.3 | Sonnet |
| 2 | **Curator** (Куратор) | Дедуплицирует и кластеризует сырые сигналы. Превращает их в **формализованные гипотезы** с базовой структурой (предмет, источник, релевантность отрасли). | 0.4 | Sonnet |
| 3 | **TechAnalyst** (Инженер-технолог) | Оценивает применимость гипотезы на текущем оборудовании компании. Учитывает технологические ограничения, нужные модификации, риски. | 0.3 | Sonnet |
| 4 | **MarketAnalyst** (Маркетолог) | Анализирует рыночную привлекательность: размер рынка, тренды, конкуренты, целевые сегменты, ценовое позиционирование. | 0.5 | Sonnet |

### 3.3 Жизненный цикл гипотезы

Гипотеза проходит через состояния (это и есть «8 этапов» из исходного документа, но привязанные к агентам):

```
[draft] → [signal_processed] → [tech_evaluated] → [market_evaluated] →
[economics_evaluated] → [compliance_checked] → [synthesized] →
[challenged] → [committee_ready] → [committee_decision] → [accepted | rejected | parked]
```

На каждом переходе:
- Соответствующий агент берёт гипотезу, обогащает её, выдаёт вердикт.
- Если **режим «требовать подтверждение»** включён — переход требует одобрения куратора.
- Если **автономный режим** — переход случается автоматически по правилам.
- Все промежуточные данные сохраняются (см. п. 3.6 Time Machine).

### 3.4 Tools, доступные агентам

Реализуй как `packages/tools/` с единым реестром. Каждый tool — Python-функция с Pydantic input/output, покрытая декоратором `@tool` (по аналогии с Anthropic Tools API). Агентам доступны через MCP-совместимый интерфейс.

**Базовый набор:**

- `web_search(query, source_type, date_range)` — поиск в открытом вебе через Anthropic web_search tool. `source_type` ∈ {`patents`, `scientific`, `news`, `competitors`, `general`}.
- `web_fetch(url)` — получение содержимого конкретного URL.
- `parse_pdf(file_id)` — извлечение текста и метаданных из PDF (для патентов и статей).
- `embed_text(text)` — генерация эмбеддинга через локальную модель.
- `search_knowledge_base(query, filters)` — RAG-поиск по накопленной базе гипотез, патентов, отчётов.
- `calc_margin(cost_components, market_price, scenarios)` — расчёт маржинальности с Monte Carlo для волатильности.
- `check_iso_standard(product_type, jurisdiction)` — проверка применимых стандартов (через AI-парсер открытых баз стандартов).
- `check_patent_freedom(product_description, jurisdictions)` — проверка патентной чистоты через поиск по патентным базам.
- `query_internal_data(query)` — запрос к внутренним данным компании (себестоимость, оборудование, цены поставщиков). Сначала заглушки, потом — реальные интеграции.
- `notify_user(user_id, message, urgency)` — отправка уведомления в Битрикс24 / Telegram / email.
- `commit_decision(hypothesis_id, decision, rationale)` — фиксация решения по гипотезе.

**Важно:** каждый вызов tool логируется. По любому событию в системе можно посмотреть «какие инструменты использовались» — это ядро трассируемости.

### 3.5 AI-парсер как переходное решение

Поскольку у тебя **пока нет официальных API** к Scopus, Reaxys и проф. базам, реализуй **AI-парсер** как универсальный инструмент. Это критически важная архитектура.

**Как работает AI-парсер:**

1. Источник описан в `sources` таблице с полями: `name`, `url_pattern`, `source_type`, `parsing_strategy`, `selectors_hint`, `auth_config`, `rate_limit`, `is_active`, `api_endpoint` (опционально).
2. Если `api_endpoint` пустой → используется AI-парсер: Scout-агент с tools (`web_search`, `web_fetch`) самостоятельно ищет, открывает страницы, извлекает структурированные данные, нормализует под нашу схему.
3. Если `api_endpoint` заполнен → используется прямой API-вызов (более надёжный и дешёвый).
4. **Каждый источник имеет «хвостик API»** — поле `api_endpoint` + `api_auth` + `api_schema_mapping`. Когда у компании появится подписка на Scopus, нужно лишь заполнить эти поля и переключить флаг `prefer_api: true`. Никакого переписывания логики.

**Список источников для ЛКМ-домена (стартовый):**
- **Патенты:** EspaceNet (EPO, открытый), Google Patents, ФИПС (Россия), USPTO. Все через AI-парсер на старте.
- **Научные публикации:** Google Scholar, arXiv, ChemRxiv, PubMed (для биоразлагаемых ЛКМ), ResearchGate. Подготовить заглушку под Scopus/Web of Science API.
- **Отраслевые новости:** Coatings World, Paint & Coatings Industry, European Coatings, ЛКМ-портал (rus). Через RSS, где есть, иначе AI-парсер.
- **Рынки сырья:** ICIS, Tecnon OrbiChem (закрытые), на старте — открытые источники типа Trading Economics, биржи. Заглушки под платные API.
- **Конкуренты:** сайты PPG, AkzoNobel, Sherwin-Williams, BASF Coatings, Hempel + российских (Эмпилс, Tikkurila Russia, Расцвет). AI-парсер с расписанием раз в неделю на новости/каталоги.
- **Стандарты:** ISO/TC 35, ГОСТ Р, REACH (ECHA), TSCA. AI-парсер на изменения.
- **Тренды:** Google Trends API (бесплатный pytrends), без полного API закрыта только глубокая аналитика.

Каждый источник имеет UI-карточку, где видно: статус, последний успешный парсинг, среднее количество сигналов в день, стоимость токенов на парсинг. Можно вручную запустить, отключить, изменить расписание.

### 3.6 Time Machine (исторические оценки)

Каждое решение агента сохраняется как **immutable событие**. Гипотеза — не «текущее состояние», а **последовательность снимков**.

Это даёт:
- Можно видеть «как мы оценивали эту гипотезу 3 месяца назад» и сравнить с текущей оценкой.
- Если переоценка дала противоположный результат — система подсвечивает «инверсию» и просит куратора подтвердить.
- При изменении внешних условий (цена сырья, новый патент конкурента) — автоматическая переоценка гипотез, которые от этого зависят, с сохранением старых оценок.

Реализация: таблица `hypothesis_evaluations(id, hypothesis_id, agent_id, run_id, evaluated_at, snapshot jsonb, prev_evaluation_id, delta jsonb)`. Через materialized view собираем «текущее состояние».

### 3.7 Memory Resurrection

Отвергнутые гипотезы не удаляются. У каждой `rejected` гипотезы есть **условия возрождения** — структурированные триггеры:

```yaml
resurrection_triggers:
  - type: "price_change"
    target: "soybean_oil"
    operator: "<"
    value: 800
    unit: "USD/ton"
  - type: "patent_expiry"
    patent_id: "US12345678"
    expiry_after: "2027-06-01"
  - type: "regulation_change"
    keyword: "biodegradable mandate EU"
```

Раз в день фоновый процесс сканирует триггеры. Если хотя бы один сработал — гипотеза автоматически перемещается в статус `to_review` с пометкой «возвращена из архива по триггеру X», куратор получает уведомление.

### 3.8 Counterfactual Engine

Кнопка «А что если…» на любой гипотезе или на дашборде в целом. Открывает форму:

```
Если бы:
- Цена пигмента TiO2 упала на ___%
- У нас появилось оборудование типа ___
- Появился доступ к технологии ___
- Изменился стандарт ___
```

Система берёт текущие гипотезы (включая отвергнутые), пересчитывает их с новыми условиями, показывает diff: какие из отвергнутых стали бы привлекательными, какие активные — наоборот, обесцениваются. Это **не сохраняется** в основной поток (это «параллельная вселенная»), но можно сохранить как именованный сценарий.

### 3.9 War Room Mode

Активируется на конкретной гипотезе кнопкой `Активировать War Room`. Эффект:
- Все агенты переходят на максимальный приоритет именно этой гипотезы.
- Пайплайн ускоряется: агенты используют Opus вместо Sonnet, увеличиваются лимиты токенов, отключается дедупликация, добавляется параллельный сбор по 3× источников.
- На дашборде появляется индикатор «WAR ROOM ACTIVE: hypothesis X».
- Каждые N часов система пушит обновление куратору.
- Автоматически отключается через заданное время (default 72 часа) или вручную.

### 3.10 Devil's Advocate

Перед статусом `committee_ready` каждая гипотеза обязательно проходит через DevilsAdvocate. Этот агент имеет особый системный промпт: «Твоя единственная задача — найти, почему эта гипотеза провалится. Атакуй её всеми возможными способами: технически, маркетингово, экономически, юридически. Будь беспощаден, но честен.»

Результат — структурированный список **контр-аргументов** с категориями и серьёзностью. Synthesizer затем интегрирует их в финальный отчёт как раздел «Риски и контр-аргументы», чтобы комитет видел не только плюсы.

### 3.11 Регулировка температур и параметров агентов

Полный экран **Agent Studio** (см. модуль 4.6). Там для каждого агента можно настроить:
- **Temperature** (слайдер 0.0 – 1.0, шаг 0.05).
- **Max tokens** (output limit).
- **Model** (Sonnet / Opus / тот, который добавится в будущем).
- **System prompt** (с историей версий и a/b-тестами).
- **Tools** (список разрешённых tools).
- **Auto-confirm** (требовать подтверждение человека или нет).
- **Cost limit per run** (лимит токенов на одну гипотезу).
- **Schedule** (если агент работает в фоне — расписание).

**Пресеты режима мышления** для всей системы (одной кнопкой меняет температуры всех агентов):
- `Conservative` — низкие температуры, агрессивный отсев. Меньше гипотез, выше качество.
- `Balanced` — дефолт.
- `Explorer` — высокие температуры аналитиков, мягкий отсев. Больше гипотез, ниже точность.
- `Maverick` — максимально широкий поиск, для брейнштормов раз в квартал.

История изменения настроек логируется (audit trail для понимания «почему агент стал хуже работать»).

## 4. Функциональные модули

### 4.1 Mission Control (главный экран)

**Это «лицо» продукта.** Заходишь и за 5 секунд понимаешь состояние системы.

**Layout:**
- Верхняя полоса: тумблер `Автономный / С подтверждениями`, индикатор War Room (если активен), глобальный счётчик стоимости за сутки/месяц.
- **Левая колонка (узкая):** живой лог активности — «Scout читает патент EP3456789…», «Economist пересчитывает гипотезу #142…». Лента обновляется в реальном времени через Supabase Realtime / WebSocket.
- **Центральная зона:** **воронка гипотез** — visual funnel со стадиями (см. 3.3). На каждой стадии — счётчик, список топ-3 гипотез, кнопка «развернуть». Можно перетаскивать карточки между стадиями (drag-and-drop) — при перетаскивании запускается переоценка соответствующим агентом.
- **Правая колонка:** «требуют внимания» — список гипотез, где система ждёт решения куратора (если включён режим подтверждений), либо где обнаружены инверсии оценок, либо сработали триггеры возрождения.
- **Низ:** компактный график динамики за 30 дней — количество новых сигналов, новых гипотез, отвергнутых, продвинутых, потраченных токенов.

Цветовое кодирование строгое: зелёный — продвинуто/принято, янтарный — требует внимания, красный — отвергнуто/риск, серый — нейтрально.

### 4.2 Process Map (Карта процесса)

**Отдельная вкладка**, ключевая для понимания, как всё работает. Реализация на **React Flow**.

**Что показывает:**
- Все 9 агентов как **узлы** с иконками и названиями.
- Связи между ними — **направленные стрелки** (поток гипотез).
- Каждый узел кликабелен → открывается боковая панель с информацией: что делает агент, какие tools использует, текущая нагрузка (сколько гипотез в очереди), средняя длительность работы, средняя стоимость, последние 5 решений.
- Линии **анимируются**, когда по ним проходит реальная гипотеза (lightweight pulse-эффект). При клике на анимирующуюся линию видишь, какая гипотеза сейчас передаётся.
- Источники данных — **отдельная панель слева** на той же диаграмме: каждый источник как узел, идущий в Scout. Клик по источнику → его статус, расписание, метрики.
- Внешние интеграции (Битрикс24, Obsidian) — **узлы справа**, куда уходят данные из Synthesizer.
- Можно переключать **режим карты:** «Schema» (статичная архитектура) и «Live» (реальный поток сейчас) и «Time Range» (агрегация за период — толщина линий = объём гипотез прошедших по ним).

Это не просто красивая картинка. Это **живая документация системы**, которая никогда не устаревает, потому что генерируется из реального кода и БД.

### 4.3 Hypothesis Workbench (Карточка гипотезы)

Главный экран работы с конкретной гипотезой. Layout — **трёхколоночный**, как в IDE.

**Левая колонка** — навигация и связи:
- Hypothesis DNA — список из ~30 структурированных метрик (см. 4.3.1).
- Связанные гипотезы (вычисляются по cosine similarity эмбеддингов).
- Связанные патенты, статьи, продукты конкурентов.
- История переходов по статусам (Time Machine).

**Центральная колонка** — содержание:
- Заголовок, краткое описание, текущий статус (с возможностью вручную перевести), куратор.
- Tabs: `Сводка` / `Технико` / `Рынок` / `Экономика` / `Стандарты` / `Риски` / `История` / `Источники`.
- В каждом табе — **результат соответствующего агента** в структурированном виде: итог, цепочка рассуждений (collapsible), использованные источники с прямыми ссылками, метрики работы агента (время, стоимость).
- Над контентом — кнопка `Переоценить` — запустить агента ещё раз с обновлёнными данными.

**Правая колонка** — действия и обсуждения:
- Чат с агентами (можно задать любому агенту вопрос по этой гипотезе).
- Комментарии команды (тред между Исследователем, Технологом, Маркетологом — обычные человеческие комментарии).
- Действия: `Продвинуть на следующий этап`, `Отвергнуть`, `Парковать`, `Активировать War Room`, `Запустить Counterfactual`, `Экспорт в Битрикс24`, `Экспорт в Obsidian`.

#### 4.3.1 Hypothesis DNA — структура

Каждая гипотеза описывается ~30 метриками в Pydantic-схеме. Метрики разбиты на блоки:

**Идентификация:** id, название, краткое описание, тип продукта, домен (ЛКМ/СОЖ/…), источник идеи.

**Технико:** сложность реализации (1–5), требуемые модификации оборудования (none/minor/major/new), готовность сырья (доступно/частично/закрытое), TRL (technology readiness level 1–9).

**Рынок:** размер рынка (в млн руб./год), темп роста (CAGR), конкурентная плотность (low/medium/high), целевые сегменты (массив), географический фокус.

**Экономика:** прогноз себестоимости, прогноз цены, маржа (%), маржа (руб/ед), объём минимальной партии для рентабельности, окупаемость инвестиций (мес).

**Риски:** общий риск-скор (1–10), патентный риск, регуляторный риск, риск волатильности сырья, технологический риск.

**Уверенность:** confidence_score (0–1) — насколько система уверена в оценках. Зависит от полноты источников. Низкая уверенность = больше человеческого внимания нужно.

**Метаданные:** дата создания, дата последней оценки, тек. куратор, статус, теги.

Все метрики — нормализованные в едином формате. Это позволяет **визуальное сравнение гипотез как ТТХ** (см. 4.4).

### 4.4 Hypothesis Compare

Выбираешь 2–5 гипотез чекбоксами в реестре → кнопка `Сравнить`. Открывается экран с **матрицей**:
- Строки — метрики (из Hypothesis DNA).
- Колонки — гипотезы.
- Цветовое кодирование разницы значений (heatmap).
- Внизу — **agentic summary** от Synthesizer: «Гипотеза А выигрывает по марже, но Гипотеза Б имеет в 3 раза меньший патентный риск. Гипотеза В — самая сбалансированная, но требует серьёзной модификации оборудования.»

### 4.5 Knowledge Graph

Ещё одна **отдельная вкладка**. Реализация на **react-force-graph-2d** или **Cytoscape.js**.

**Что в графе:**
- **Узлы:** гипотезы, патенты, научные статьи, материалы/сырьё, технологии, конкуренты, продукты, стандарты.
- **Рёбра:** «использует материал», «конкурирует с», «развивает идею», «нарушает патент», «соответствует стандарту», «ссылается на», «производится тем же оборудованием».
- Разные **цвета** для разных типов узлов, **размер** узла = его «важность» (degree centrality).

**Возможности:**
- Поиск и переход к узлу.
- Фильтры: тип узла, дата, домен, статус.
- Кластеризация: автоматическое определение кластеров технологий (community detection через Louvain).
- При клике на узел — карточка с подробностями + кнопка «открыть гипотезу».
- **Path finder:** «как связана эта гипотеза с этим конкурентом?» — система ищет кратчайший путь в графе.
- **Цепочки рассуждений:** «эта гипотеза появилась из этой статьи, которая ссылается на этот патент, который владеет этот конкурент».

Граф генерируется по мере накопления данных. Минимально жизнеспособный граф появляется после ~50 обработанных сигналов.

### 4.6 Agent Studio

Экран настройки агентов (см. 3.11). Левый сайдбар — список агентов. По клику — справа форма с настройками. Для системного промпта — Monaco editor (как в VS Code).

**Особое внимание:** каждое изменение системного промпта/параметров создаёт новую **версию** агента. Можно сравнить производительность v1 vs v2 (среднее время, стоимость, % правильных решений по обратной связи кураторов). Можно откатиться на любую версию.

**A/B-тестирование:** можно запустить агента в режиме «параллельного тестирования» — 50% гипотез обрабатывает v1, 50% — v2, через неделю смотришь статистику.

Сверху — пресеты режимов мышления одной кнопкой (Conservative / Balanced / Explorer / Maverick).

### 4.7 Sources Manager

Управление источниками (см. 3.5). Список с фильтрами по типу и статусу. У каждого источника:
- Карточка статуса: активен/приостановлен, последний парсинг, успех/провал, среднее количество сигналов.
- Настройки: расписание, rate limit, зона ответственности.
- Поля для будущих API: `api_endpoint`, `api_auth`, `prefer_api`. Когда заполнено — автоматический переключение с AI-парсера на API.
- Тестовый запуск: «прогнать парсер сейчас и показать результат» (без сохранения).
- Метрики стоимости: сколько токенов потрачено на парсинг этого источника за месяц.

Стартово создай записи для всех источников из 3.5.

### 4.8 Reports & Committee

Подготовка к комитету и проведение голосования.

- **Подборка для комитета:** выбираешь N готовых гипотез (`committee_ready`) → формируешь подборку.
- **Автогенерация документов:** Synthesizer генерирует:
  - Executive summary (Markdown / PDF).
  - Презентация (через генерацию PPTX из шаблонов).
  - Excel-таблица сравнения гипотез.
- **Комитетная сессия:** запланированное голосование по каждой гипотезе. Каждый член комитета (роль `committee_member`) заходит и голосует: `Запустить пилот` / `Отложить` / `Отклонить` / `Запросить доп. данные`. Каждый голос с обязательным комментарием.
- **Результат:** финальное решение комитета фиксируется, гипотеза переходит в `accepted` / `parked` / `rejected`. Если `accepted` — автоматически создаётся задача в Битрикс24 (при включённой интеграции).

### 4.9 Search & Command Palette

- Глобальный поиск по всему: гипотезы, агенты, источники, патенты, заметки. С учётом семантики (через эмбеддинги).
- Command palette `Cmd+K` (Linear-like): «open hypothesis #142», «activate war room», «set conservative mode», «add source RSS …», «show agent studio».
- Это **обязательно** для скорости работы.

### 4.10 Notifications & Notion / Obsidian / Bitrix24

- Уведомления внутри системы (правый верхний угол).
- Email + Telegram + push (web push).
- В Битрикс24 — задачи и сообщения в чат.
- В Obsidian — Markdown-заметка для каждой принятой гипотезы создаётся в watched folder с frontmatter:
  ```yaml
  ---
  type: hypothesis
  id: 142
  domain: ЛКМ
  status: accepted
  created: 2026-04-15
  margin_pct: 38
  risk_score: 4
  tags: [биоразлагаемый, соевое масло, B2B]
  ---
  ```
- Граф гипотез также экспортируется в Obsidian как набор связанных заметок (можно открывать его граф знаний прямо в Obsidian).

### 4.11 Admin

- Управление пользователями и ролями.
- Управление доменами (ЛКМ + 9 других — включение/отключение, специфичные настройки на домен).
- Бюджеты на токены (месячный лимит, алерты при превышении).
- Журнал аудита (все изменения настроек, все принятые решения).
- API ключи к внешним сервисам (зашифрованные, masked в UI).
- Бэкапы и экспорт данных.

## 5. Дизайн-система и UX

### 5.1 Палитра

Не унылый «корпоративный синий», но и не радуга. Тёмная тема — основная (она лучше передаёт ощущение «исследовательской лаборатории», когда работаешь долго). Светлая — опциональная.

- **Background:** `#0A0A0F` (основной), `#13131A` (поверхности), `#1C1C25` (приподнятые элементы).
- **Text:** `#E4E4E7` (основной), `#A1A1AA` (вторичный), `#71717A` (приглушённый).
- **Borders:** `#27272A`.
- **Accent:** глубокий аметистовый `#7C3AED` (Compass-фиолетовый, заменяемый CSS-переменной).
- **Семантика:** зелёный `#10B981`, янтарный `#F59E0B`, красный `#EF4444`, синий `#3B82F6` (info).
- **Светлая тема** — инвертированная, с теми же акцентами.

### 5.2 Типографика

## 6. Безопасность и приватность

- **Все промпты, ответы LLM, экстракт данных** — хранятся encrypted at rest.
- **API-ключи** для внешних сервисов — в Supabase Vault или HashiCorp Vault, никогда в коде.
- **Row Level Security (RLS)** в Supabase — пользователь видит только то, что ему положено по роли.
- **Audit log** — все изменения, все решения, все запуски агентов.
- **Rate limiting** на стороне API (Redis-based).
- **Не отправлять во внешние API** конкретные внутренние данные компании (себестоимость, поставщики) без явной настройки. По умолчанию — внутренние данные обрабатываются только локально (или через Anthropic API с пометкой «не использовать для обучения»).
- **MFA** для всех пользователей, кроме разработки.
- **Бэкапы** — ежедневно, с проверкой восстановления раз в квартал.

## 7. Roadmap (порядок реализации)

Двухфазная стратегия (как мы обсуждали):

### Фаза 1 — Монолитный MVP (1.5 месяца)

Цель — рабочая система с **жёсткими пайплайнами**, без LangGraph-оркестрации. Каждый «агент» — это просто async-функция, которая принимает гипотезу и возвращает обновлённую. Пайплайн жёстко закодирован: 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8.

**Что входит:**
- Монорепо, CI/CD, Docker, Supabase setup.
- Auth (Supabase Auth), 5 ролей, RBAC.
- Layout, дизайн-система, command palette, тосты.
- Mission Control (упрощённый: воронка + лента активности + список «требуют внимания»).
- Hypothesis Workbench (без всех тонкостей, базовая карточка).
- Sources Manager с AI-парсером (3 типа источников: патенты, новости, статьи; для ЛКМ).
- 4 базовых агента: Scout, Curator, TechAnalyst, Synthesizer (без Economist, MarketAnalyst, Compliance, DevilsAdvocate — для них заглушки).
- AI-Gateway с логированием и подсчётом стоимости.
- Базовая Time Machine (просто хранение снимков).
- Тумблер «автономный / с подтверждением» (общий, на уровень всей системы).
- Уведомления в email и Telegram.

**Не входит в Фазу 1:** Process Map, Knowledge Graph, Counterfactual, War Room, Memory Resurrection, Agent Studio (только базовые настройки), Compare, Reports & Committee (упрощённо).

**Критерий завершения:** запустил систему, она за неделю работы выдала ≥10 проработанных гипотез по ЛКМ, юзабельных для обсуждения с экспертом.

### Фаза 2 — Агентная архитектура (1.5 месяца)

Цель — превратить жёсткий пайплайн в LangGraph-агентов с инновационными фичами.

**Что входит:**
- Миграция на LangGraph: каждый агент — нода в graph, оркестратор — отдельный агент.
- Добавление агентов: Economist, MarketAnalyst, ComplianceOfficer, DevilsAdvocate.
- **Process Map** (полноценный, с live-режимом).
- **Knowledge Graph** (с базовой кластеризацией).
- **Hypothesis DNA** (полные 30 метрик).
- **Compare** (матрица сравнения).
- **Agent Studio** (полноценный, включая A/B и пресеты).
- **War Room Mode**.
- **Counterfactual Engine**.
- **Memory Resurrection** (триггеры и фоновое сканирование).
- **Reports & Committee** (генерация PPTX, голосование).
- Интеграция с **Битрикс24** (двусторонняя).
- Интеграция с **Obsidian** (через файловую синхронизацию).
- Расширение источников: добавление по 2–3 источника на каждую категорию.
- Подготовка архитектуры под другие 9 доменов (СОЖ, смазки и т.д.) — переключатель домена в UI, доменные playbook'и для агентов.

**Критерий завершения:** система выдаёт ≥5 готовых для комитета гипотез в неделю с обоснованиями уровня качества «чем хуже, чем хороший аналитик».

### Фаза 3 — Расширение и зрелость (по мере появления потребностей)

class Hypothesis(BaseModel):
    id: UUID
    title: str
    short_description: str
    long_description: str | None = None
    domain: Literal["lkm", "soj", "lubricants", "anticor", "sealants",

    technical: TechnicalProfile | None = None
    market: MarketProfile | None = None
    economics: EconomicsProfile | None = None
    risks: RisksProfile | None = None

    confidence_score: float = 0.0  # 0..1
    overall_score: float | None = None  # композитный, считается Synthesizer

    source_signals: list[UUID] = []  # связанные сигналы

    war_room_active: bool = False

    resurrection_triggers: list[ResurrectionTrigger] = []

    created_at: datetime
    updated_at: datetime
    last_evaluated_at: datetime | None = None
